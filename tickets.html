What is a Ticket System?

Think of it like a team to-do list on steroids.

A ticket is simply a task that needs to get done. Every bug fix, new feature, or improvement starts as a ticket. The ticket system makes sure nothing gets lost, everyone knows who is working on what, and work flows smoothly from start to finish.

Think of it like a restaurant kitchen:

  • 🍽️ A customer orders a dish (this is a ticket)
  • 📝 The order is written down with all the details (ticket context)
  • 👨‍🍳 The chef cooks each step in order (ticket executor)
  • 🧑‍💼 The manager tracks all orders, prioritizes, and makes sure everything runs (ticket manager)

Why this matters for you

Instead of juggling emails, messages, and sticky notes, every task has a single home. You always know:

  • ✅ What you should be working on
  • ✅ What the quality bar is
  • ✅ Who to ask when stuck
  • ✅ What happens after you finish

How the Three Pieces Fit Together

Three AI "skills" work as a team to take a task from idea to completion. Here is the flow in everyday language:

1

Ticket Context

The "what and why" of the task.

Loads the ticket, checks that everyone agrees on what done means, and makes sure the task is ready to start.

2

Ticket Executor

The "how" — does the actual work.

Follows the plan step by step, checks for quality after each step, and handles urgent "drop everything" tasks.

3

Ticket Manager

The "big picture" organizer.

Creates tickets, moves them through stages, tracks progress across all projects, and generates team overviews.

The full journey: Got an idea? → Write it as a ticket → Plan the work → Do the work → Check it → Ship it

Skill 1

ticket_context

"Load the ticket and make sure we all agree on what needs to happen."

What it does

Before anyone starts writing code, this skill finds the ticket file, reads it, and checks that the acceptance criteria (the list of things that must be true for the task to be considered done) are clear and complete. If something is missing or unclear, it asks for clarification before work starts — saving everyone time.

How it works (step by step)

  1. 1 You give it a ticket ID like WOW-042
  2. 2 It finds the ticket file in the correct folder
  3. 3 It checks the acceptance criteria against our quality bar
  4. 4 If something would break our quality rules, it stops and asks you to fix it

What "production-ready" means

Before any code is accepted, it must pass these checks:

  • ❌ No fake/test data pretending to be real
  • ✅ Errors are handled gracefully (no crashes)
  • ✅ We can see what the code is doing (logging/monitoring)
  • ✅ Security is not an afterthought
  • ✅ Edge cases (weird inputs, unusual situations) are handled
  • ✅ Failure scenarios are tested

In plain English: Think of this as a quality gatekeeper. Before a builder starts building, someone checks that the blueprints are complete and meet building codes. This skill does that — automatically.

Skill 2

ticket_executor

"Follow the plan, one step at a time, and stop if something goes wrong."

What it does

This is the "doer." It takes a plan and executes it in phases. After each phase it checks that everything is working (telemetry = measuring that the code does what it should). It will not move to the next step until the current step is verified.

Commands you can use

/implement_plan

Start executing the full plan

/execute_phase

Run just the next phase

/skip_phase

Skip a phase you do not need

🚨

Enforcement Tickets — The "Drop Everything" Button

Sometimes a critical issue comes up that is more important than anything else. An enforcement ticket lives in a special folder and overrides all other work. While it is open, all other tasks must pause until it is resolved. This is how we handle emergencies without losing track of regular work.

In plain English: Imagine renovating a house room by room. You finish the kitchen, check that everything works (lights, plumbing), then move to the bathroom. If there is a gas leak (enforcement ticket), you stop everything immediately and fix it first.

Skill 3

ticket_manager

"The organizer — creates, tracks, and manages tickets across all projects."

What it does

This is the control center. It handles everything from creating a new ticket to moving it through the stages of work, assigning people, and making sure nothing falls through the cracks. It works across all three projects (WayOfMono, WayOfWork, Opticat) at the same time.

How tickets are named

Every ticket gets a unique name so you can find it instantly. The format is:

<PROJECT>-<NUMBER>-<SHORT-DESCRIPTION>.md

Real examples:

WOMONO-044 → WayOfMono project, ticket #44
WOW-001 → WayOfWork project, ticket #1
OPT-001 → Opticat project, ticket #1

Tip: The description is always in UPPERCASE with dashes instead of spaces, like ADD-LOGIN-BUTTON

Where tickets live (storage)

All tickets are stored as simple files in the f-rr-d repository. Each project has its own folder:

WOMONO tickets thoughts/wayofmono/shared/tickets/
WOW tickets thoughts/wow/shared/tickets/
OPT tickets thoughts/opticat/shared/tickets/

Your personal tickets: thoughts/<project>/<your-name>/tickets/

🚨 Emergency tickets: thoughts/<project>/enforcement-ticket/

Blank template: thoughts/shared/tickets/ticket-template.md

The life of a ticket (status flow)

Every ticket moves through these stages. Think of it like a package tracking system:

📥 Backlog → 📋 Planned → ✅ Ready → 🔧 In Progress → 👀 Submitted for Review → 📖 In Review → 👍 Approved → ✅ Done
If something needs fixing: 🔄 Changes Requested → back to → 🔧 In Progress or 🚫 Blocked

Useful commands

Type these in the AI chat to manage tickets:

/work WOW-042

Start working on ticket WOW-042

/complete WOW-042

Mark it as done (or submit for review)

/sync team

See what everyone is working on

Create a new ticket (guided wizard)

Ticket information fields

Every ticket has these fields at the top (frontmatter):

📌 title What is this task?
🏷️ type Feature/Bug/Fix
⚡ priority How urgent?
📊 status Where in the flow?
👤 assignee Who is doing it?
📝 reporter Who created it?
🔗 pr_url Link to the code changes

Quick reference: Types & Priorities

Types of tickets:

  • ✨ Feature — A new capability
  • 🐛 Bug — Something broken
  • 🧹 TechDebt — Clean up/improve code
  • 📦 Epic — A big multi-step project
  • 💡 Improvement — Make something better

Priority levels:

  • 🔴 Critical — Must fix now (blocks everything)
  • 🟠 High — Important, do soon
  • 🟡 Medium — Normal priority
  • ⚪ Low — Nice to have

f-rr-d (förråd) — The Central Brain

Swedish for "storage" — this is where all tickets, plans, and knowledge live

f-rr-d (pronounced "förråd", Swedish for "storage") is a git repository that acts as the single source of truth for everything: tickets, plans, research notes, and personal todos. It is cloned inside each project as a folder called thoughts/.

Important rule: APPEND ONLY

You can add new files, but never delete, rename, or edit existing ones. This keeps a complete history of every decision and idea — like a digital diary that never forgets.
